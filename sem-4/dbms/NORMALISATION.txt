Q.) Define Normalization. Explain 1NF, 2NF and 3NF with suitable example 



A. Introduction to Normalization and Anomalies

Normalization is closely related to the idea of efficiency and the elimination of redundancy. Where redundancy exists, there is a risk that certain operations will create data anomalies. The term anomaly is used to describe something that is not in line with expectations. In a relational database, we expect that all of the data stored will be correct and internally consistent.

When a database is not normalized, we run into three specific types of anomalies. Imagine a poorly designed single table storing [Student_ID, Student_Name, Course, Instructor, Instructor_Phone]:

Update Anomaly: If an instructor changes their phone number, you have to update it in hundreds of rows (every row for every student taking their course). If you miss even one row, the data is inconsistent.

Insertion Anomaly: You cannot add a newly hired Instructor and their Phone Number to the database until at least one student enrolls in their course (because Student_ID is required for the row).

Deletion Anomaly: If the only student enrolled in an advanced course drops it, deleting that student's row entirely erases the fact that the course, the instructor, and the instructor's phone number even exist.

To fix these anomalies, we pass the data through "Normal Forms."



B. The Normal Forms (The "Rules" of Database Design)

Let's look at a messy, Unnormalized table to start:
Student_Records Table
| Roll_No (PK) | Name | Department | Dept_Head | Subjects |
| :--- 	       | :--- | :--- 	   | :---      | :---     |
| 101          | Alex | CompSci    | Dr. Smith | Java, SQL, Python |
| 102          | Sam  | Physics    | Dr. Jones | Mechanics, Optics |

1. First Normal Form (1NF): (Atomic Values)
The Rule: A table is in 1NF if every column holds a single, indivisible (atomic) value, and there are no repeating groups or arrays.

The Problem: In our table, the Subjects column holds multiple values separated by commas. This makes it a nightmare to search for all students taking "SQL".
The Fix: We split the comma-separated values into their own rows.

Table in 1NF:
| Roll_No (PK) | Subject (PK) | Name | Department | Dept_Head |
| :---         | :---         | :--- | :---       | :---      |
| 101          | Java         | Alex | CompSci    | Dr. Smith |
| 101          | SQL          | Alex | CompSci    | Dr. Smith |
| 102          | Optics       | Sam  | Physics    | Dr. Jones |

Note: Because Roll_No 101 repeats, our Primary Key is now a Composite Key: (Roll_No + Subject).

2. Second Normal Form (2NF): (No Partial Dependency)
The Rule: A table is in 2NF if it is in 1NF and all non-key columns depend on the entire primary key, not just part of it. (This only applies if you have a composite primary key).

The Problem: Look at our 1NF table. Our primary key is (Roll_No, Subject). But does a student's Name or Department depend on the Subject? No. Alex's name is Alex regardless of the subject. Name only depends on Roll_No. This is a partial dependency, which causes massive redundancy.
The Fix: Split the table into two. One for the Student details, one for the Subjects they take.

Tables in 2NF:
Student_Table:
| Roll_No (PK) | Name | Department | Dept_Head |
| :---         | :--- | :---       | :---      |
| 101          | Alex | CompSci    | Dr. Smith |
| 102          | Sam  | Physics    | Dr. Jones |

Enrollment_Table:
| Roll_No (PK/FK) | Subject (PK) |
| :---            | :--- |
| 101             | Java |
| 101             | SQL  |

3. Third Normal Form (3NF): (No Transitive Dependency)
The Rule: A table is in 3NF if it is in 2NF and no non-key column depends on another non-key column. All non-key columns must depend strictly on the primary key.

The Problem: Look at the Student_Table from 2NF. Dept_Head tells us who runs the department. But Dept_Head actually depends on the Department column, not the Roll_No. (Roll_No -> Department -> Dept_Head). This is a transitive dependency. If Alex changes majors, we shouldn't have to worry about updating Dr. Smith's data.
The Fix: Move the department details into their own table.

Tables in Final 3NF:
Student_Table:
| Roll_No (PK) | Name | Department (FK) |
| :--- 	       | :--- | :--- |
| 101          | Alex | CompSci |
| 102          | Sam  | Physics |

Department_Table:
| Department (PK) | Dept_Head |
| :---		  | :---      |
| CompSci 	  | Dr. Smith |
| Physics 	  | Dr. Jones |

(The Enrollment_Table from step 2 remains untouched).

Summary
By reaching 3NF, we have successfully divided a large, redundant table into smaller, related tables linked by Primary and Foreign keys.

1NF eliminated repeating lists.

2NF eliminated data tied to only part of a key.

3NF eliminated data tied to non-keys.
As a result, update, insertion, and deletion anomalies are completely prevented!

You've got a great grasp on the first three normal forms. Let's keep climbing the normalization ladder. Picking up right where you left off, here is how Boyce-Codd Normal Form (BCNF) and Fourth Normal Form (4NF) fit into database design.

 C. Boyce-Codd Normal Form (BCNF)

Also known as 3.5NF.

The Rule: A table is in BCNF if it is in 3NF, and for *every* functional dependency X -> Y, X is a superkey. In simpler terms: If any attribute determines another attribute, it must be a unique identifier (key) for the whole row.

The Problem: 3NF has a slight blind spot. It allows a non-key column to determine a *part* of a composite primary key. This usually happens when a table has overlapping candidate keys.

Let's look at a schedule table where a student takes a course, and each course has multiple professors, but each professor only teaches *one* specific course.

Student_Advising Table (Violates BCNF):
*Primary Key: (Student_ID, Course)*

| Student_ID | Course | Professor |
| --- | --- | --- |
| 101 | Database | Dr. Turing |
| 102 | Database | Dr. Codd |
| 101 | Algorithms | Dr. Lovelace |

* The Dependency: `Professor` -> `Course`. (Because Dr. Turing *only* teaches Database).
* The Violation: `Professor` is determining `Course` (which is part of our Primary Key), but `Professor` is not a superkey (Dr. Turing could teach multiple students, so his name alone doesn't uniquely identify a row). If Dr. Turing stops teaching Database and switches to Networks, we face an update anomaly.

The Fix: Split the table so that the determinant (`Professor`) becomes a valid primary key in its own table.

Tables in BCNF:

Student_Professor Table:

| Student_ID (PK) | Professor (PK) |
| --- | --- |
| 101 | Dr. Turing |
| 102 | Dr. Codd |
| 101 | Dr. Lovelace |

Professor_Course Table:

| Professor (PK) | Course |
| --- | --- |
| Dr. Turing | Database |
| Dr. Codd | Database |
| Dr. Lovelace | Algorithms |

 D. Fourth Normal Form (4NF)

(No Multi-Valued Dependencies)

The Rule: A table is in 4NF if it is in BCNF and contains no multi-valued dependencies. A multi-valued dependency occurs when one primary key determines multiple independent attributes, forcing you to create a massive combination of rows.

The Problem: Imagine we want to track a student's technical skills and their hobbies. Skills and hobbies have absolutely nothing to do with each other. If we put them in the same table, we have to create a Cartesian product (every possible combination) just to keep the data accurate.

Student_Profile Table (Violates 4NF):
*Primary Key: (Roll_No, Skill, Hobby)*

| Roll_No | Skill | Hobby |
| --- | --- | --- |
| 101 | Python | Chess |
| 101 | Python | Guitar |
| 101 | C++ | Chess |
| 101 | C++ | Guitar |

* The Violation: Roll_No 101 has two skills and two hobbies. Because they are independent, adding a *third* skill (like Java) means we have to add two new rows (Java + Chess, Java + Guitar) just to maintain consistency. This is a multi-valued dependency: `Roll_No` ->-> `Skill` and `Roll_No` ->-> `Hobby`.

The Fix: Isolate the independent multi-valued attributes into their own separate tables.

Tables in 4NF:

Student_Skills Table:

| Roll_No (PK) | Skill (PK) |
| --- | --- |
| 101 | Python |
| 101 | C++ |

Student_Hobbies Table:

| Roll_No (PK) | Hobby (PK) |
| --- | --- |
| 101 | Chess |
| 101 | Guitar |

 Summary of the Higher Forms

* BCNF stepped in to fix the edge case where a non-key attribute was secretly dictating a prime key attribute.
* 4NF stepped in to untangle completely independent lists of data that were cross-multiplying and creating massive row bloat.
