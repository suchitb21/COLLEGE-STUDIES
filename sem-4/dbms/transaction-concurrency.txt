Q.) What is transaction ?

A.) A transaction refers to a sequence of one or more operations ( such as read, write, update, or delete  and [ COMMIT , ROLLBACK ] ) performed on the database as a single logical unit of work.

A transaction ensures that either all the operations in the sequence are successfully executed ( committed ) or none of them take effect ( roll back ) 

Transaction are designed to maintain the integrity, consistency and reliability of the database, even in the case of system failures or concurrent access.

DBMS transactions must be atomic, consistent, isolated and durable. If the DB was in inconsistent state before a transaction, it would remain in the inconsistent state after the transaction.

Q.) ACID Properties of Transaction
A.) Transaction in DBMS must ensure data is accurate and reliable. They follow four key ACID properties:

1.) Atomicity : A transaction is all or nothing. If any part fails, the entire transaction is rolled back. Example : While transferring money, both debit and credit must succeed. If one fails, nothing should change.

2.) Consistency : A transaction must keep the database in a valid state, moving it from one consistent state to another

3.) Isolation : Transaction run independently. One transaction's operations should not affect another's intermediate steps. Example : 2 users withdrawing from the same account must not interfere with each other's balance updates.

4.) Durability: Once a transaction is committed, it's changes stay even if the system crashed. Example : After a successful transfer, the updated balance remains safe despite a power failure.


Q.) What is serializable schedule ?
A.) A transaction schedule is serializable if its outcome is equal to the outcome of its transactions executed serially ( i.e sequentially without overlapping in time. ). A serializable schedule always leaves the database in a consistent state. A serial schedule is always a serializable schedule because a new transaction only starts when the older one has finished execution.


Q.) What is conflict serializability ?
A.) Conflict serializability checks if a schedule of transactions can be transformed into a sequence where transactions are executed serially, while keeping the same results. This type of serializability focuses on the order of conflicting operations.

Two operations are said to be conflicting if the belong to different transaction, operate on same data and at least one of them is a write operation. 

ADVANTAGE: DATA INTEGRITY, CLEAR RULES, EFFICIENT EXECUTION, STRONG ISOLATION, DETECTABLE ISSUES ( using precedence graph )
DISADVANTAGE: RESTRICTIVE, COMPLEXITY, PERFORMANCE OVERHEAD, DEADLOCK POTENTIAL, NA


Q.) WHAT IS VIEW SERIALIZABILITY?

View serializability checks if a schedule produces the same final results as a serial execution, regardless of the order of conflicting operations.

Key Difference from Conflict Serializability:
- Conflict Serializability: Order of conflicting operations must match serial schedule
- View Serializability: Final data values must be same as serial schedule (order doesn't matter)

---

 TWO CONDITIONS FOR VIEW SERIALIZABILITY

# Condition 1: Initial Read
If transaction T_j reads initial value of data item X in schedule S, then T_j must read initial value of X in serial schedule too.

# Condition 2: Updated Read
If transaction T_i writes X and T_j reads that value, then in serial schedule, T_i must come before T_j.

# Condition 3: Final Write
The transaction that writes the final value of X must be the same in both schedules.

 PRACTICAL EXAMPLE WITH DATABASE

Let's use a Bank Account with Initial Balance = 100


Account Table:
AccountID  Balance
----------  -------
A1         100


---

 EXAMPLE 1: View Serializable but NOT Conflict Serializable

# Schedule S:

T1: Read(X)     -- T1 reads X=100
T2: Write(X,50) -- T2 writes X=50
T1: Write(X,75) -- T1 writes X=75
T2: Read(X)     -- T2 reads X=75


# Trace Execution:

Step 1: T1 reads X=100 (initial value)
Step 2: T2 writes X=50
Step 3: T1 writes X=75 (overwriting T2's value)
Step 4: T2 reads X=75 (reads T1's updated value)

FINAL VALUE OF X = 75


---

 IS IT CONFLICT SERIALIZABLE?

# Conflict Analysis:

Operations in order:
1. T1:Read(X)
2. T2:Write(X) ──┐ CONFLICT (different transactions, same data, one is write)
                 └─ T1:Read must come BEFORE T2:Write
3. T1:Write(X) ──┐ CONFLICT (different transactions, same data, one is write)
                 └─ T2:Write must come BEFORE T1:Write
4. T2:Read(X)


# Precedence Graph:

T1 ──────> T2  (from T1:Read before T2:Write)
T2 ──────> T1  (from T2:Write before T1:Write)

CYCLE DETECTED! ❌


Result: NOT Conflict Serializable (cycle in precedence graph)

---

 IS IT VIEW SERIALIZABLE?

# Check Serial Schedule 1: T1 → T2

T1:Read(X)      -- reads initial X=100 ✓
T1:Write(X,75)  -- writes X=75
T2:Write(X,50)  -- writes X=50 (overwrites T1's value)
T2:Read(X)      -- reads X=50

FINAL VALUE = 50 ❌ (doesn't match 75)


# Check Serial Schedule 2: T2 → T1

T2:Write(X,50)  -- writes X=50
T2:Read(X)      -- reads X=50 ✓
T1:Read(X)      -- reads X=50 (NOT initial 100) ❌
T1:Write(X,75)  -- writes X=75

FINAL VALUE = 75 ✓


But T1 reads X=50 (not initial 100) ❌

# Modified Check with Blind Write:

If T2 is considered a BLIND WRITE (writes without reading):
T2:Write(X,50)
T1:Read(X)      -- reads initial X=100 (before T2's write in view)
T1:Write(X,75)
T2:Read(X)      -- reads X=75

FINAL VALUE = 75 ✓
Initial Read satisfied ✓
Updated Read: T1 writes, T2 reads → T1 before T2 ✓


Result: VIEW SERIALIZABLE ✓ (equivalent to T1 → T2 in terms of values)

---

 EXAMPLE 2: CLEAR VIEW SERIALIZABLE EXAMPLE

# Schedule S:

T1: Write(X, 20)   -- T1 writes X=20
T2: Read(X)        -- T2 reads X=20
T2: Write(Y, 40)   -- T2 writes Y=40
T1: Read(Y)        -- T1 reads Y=40


# Trace Execution:

Initial: X=100, Y=100

T1:Write(X,20)  --> X=20
T2:Read(X)      --> reads X=20
T2:Write(Y,40)  --> Y=40
T1:Read(Y)      --> reads Y=40

FINAL: X=20, Y=40


---

 IS IT CONFLICT SERIALIZABLE?

# Conflicts:

T1:Write(X,20) ──┐
T2:Read(X)    ──┘  Conflict: T1 before T2

T2:Write(Y,40) ──┐
T1:Read(Y)    ──┘  Conflict: T2 before T1

Precedence: T1 → T2 and T2 → T1
CYCLE DETECTED! ❌ NOT Conflict Serializable


---

 IS IT VIEW SERIALIZABLE?

# Serial Schedule 1: T1 → T2

T1:Write(X,20)  --> X=20
T1:Read(Y)      --> Y=100 (initial)
T2:Read(X)      --> X=20 ✓
T2:Write(Y,40)  --> Y=40

FINAL: X=20, Y=40 ✓
Initial Reads: T2 reads X=20 (written by T1) ✓
Updated Reads: T1 writes Y before T2... wait, T2 doesn't read Y
Final Writes: T2 writes Y last ✓


Result: VIEW SERIALIZABLE ✓

---

 EXAMPLE 3: NOT VIEW SERIALIZABLE

# Schedule S:

T1: Read(X)    -- reads X=100
T2: Write(X,50)
T1: Write(X,75)
T3: Read(X)    -- reads X=75


# Check View Equivalence:

Serial Schedule T1 → T2 → T3:

T1:Read(X)   --> reads 100 ✓
T1:Write(X,75)
T2:Write(X,50)
T3:Read(X)   --> reads 50

FINAL = 50, but in S final = 75 ❌


Serial Schedule T2 → T1 → T3:

T2:Write(X,50)
T1:Read(X)   --> reads 50 (not initial 100) ❌
T1:Write(X,75)
T3:Read(X)   --> reads 75


Result: NOT VIEW SERIALIZABLE ❌

---

 COMPARISON TABLE

| Aspect | Conflict Serializability | View Serializability |
|--------|--------------------------|----------------------|
| Definition | Order of conflicts must match serial schedule | Final results must match serial schedule |
| Strictness | More restrictive | More lenient |
| Detection | Precedence Graph (easy) | Complex algorithms needed |
| Example | Write(A) before Read(A) order matters | Only final value matters |
| Concurrency | Less parallelism | More parallelism possible |
| Performance | Higher overhead | Lower overhead |
| Practical Use | Most databases use this | Rarely used in practice |

---

 ADVANTAGES OF VIEW SERIALIZABILITY

✅ More Permissive: Allows schedules that conflict serializability rejects
✅ Better Concurrency: More parallel transactions possible
✅ Performance: Less locking overhead
✅ Flexibility: Blind writes (writes without reads) are allowed

---

 DISADVANTAGES OF VIEW SERIALIZABILITY

❌ Complex Detection: Hard to determine if schedule is view serializable
❌ No Efficient Algorithm: No simple precedence graph method
❌ Rarely Used: Most databases use conflict serializability instead
❌ Deadlock Issues: Still potential for circular wait conditions
❌ Less Predictable: Harder for DBAs to reason about concurrency

---

 REAL-WORLD SCENARIO

# Bank Transfer Example:


Account A = 1000
Account B = 2000

T1: Transfer 100 from A to B
T2: Transfer 50 from B to A

Schedule (View Serializable but NOT Conflict Serializable):

T1:Read(A)      --> reads 1000
T2:Read(B)      --> reads 2000
T1:Write(A,900) --> A=900
T2:Write(B,1950)--> B=1950
T1:Write(B,2100)--> B=2100
T2:Write(A,950) --> A=950

FINAL: A=950, B=2100

This is View Serializable (equivalent to some serial order)
but NOT Conflict Serializable (has cycles)


---

 SUMMARY

Conflict Serializability = Strict ordering of conflicting operations
View Serializability = Flexible, only final values matter

In Practice: Databases use Conflict Serializability because it's easier to enforce and detect, even though View Serializability could allow more concurrency.

Q.) Explain Lock-based protocols with example 

A.) Lock-based protocols are concurrency control mechanisms where transactions must acquire locks on data items before accessing them. A lock prevents other transactions from accessing the same data simultaneously, ensuring isolation and consistency.

Types of Locks:

1.) Shared Lock ( S-Lock ) : Also called a read lock. Multiple transactions can hold a shared lock on the same data item at the same time, but none can write to it.
Example : T1 and T2 both want to read Account A. Both can hold a shared lock on A simultaneously.

2.) Exclusive Lock ( X-Lock ) : Also called a write lock. Only one transaction can hold an exclusive lock on a data item. No other transaction can read or write that item until the lock is released.
Example : T1 wants to update Account A's balance. It acquires an exclusive lock — T2 must wait until T1 releases it.

Two-Phase Locking Protocol ( 2PL ) :
The most widely used lock-based protocol. It has two phases :

- Growing Phase : A transaction can acquire locks but cannot release any lock.
- Shrinking Phase : A transaction can release locks but cannot acquire any new lock.

2PL guarantees conflict serializability.

Example :


T1 : Lock-X(A) → Read(A) → Write(A) → Unlock(A) → Lock-X(B) → Read(B) → Write(B) → Unlock(B)

T1 first acquires all needed locks (growing phase), performs operations, then releases them (shrinking phase).

ADVANTAGE : Ensures serializability, Prevents dirty reads, Simple to implement
DISADVANTAGE : Can cause deadlocks, Reduced concurrency, Lock overhead



Q.) Explain Timestamp-based protocols with example

A.) Timestamp-based protocols are concurrency control mechanisms that use timestamps to determine the order in which transactions access data, without using locks. Every transaction is assigned a unique timestamp ( TS ) when it enters the system. Older transactions have smaller timestamps.

Each data item Q maintains two timestamp values :

1.) W-timestamp(Q) : The largest timestamp of any transaction that successfully wrote Q.
2.) R-timestamp(Q) : The largest timestamp of any transaction that successfully read Q.

Rules :

For a Read operation by Transaction Ti on data item Q :
- If TS(Ti) < W-timestamp(Q) → Ti is trying to read a value that has already been overwritten by a newer transaction → Transaction Ti is rolled back.
- Otherwise → Read is executed and R-timestamp(Q) = max( R-timestamp(Q), TS(Ti) )

For a Write operation by Transaction Ti on data item Q :
- If TS(Ti) < R-timestamp(Q) → A newer transaction has already read Q, so Ti's write is too late → Transaction Ti is rolled back.
- If TS(Ti) < W-timestamp(Q) → A newer transaction has already written Q → Transaction Ti is rolled back.
- Otherwise → Write is executed and W-timestamp(Q) = TS(Ti)

Example :

Suppose T1 has TS = 100 and T2 has TS = 200.
T2 reads and writes data item A first ( W-timestamp(A) = 200 ).
Now T1 tries to write A → TS(T1) = 100 < W-timestamp(A) = 200 → T1 is rolled back because a newer transaction T2 has already written A.

ADVANTAGE : No deadlocks, Ensures serializability, No waiting ( transactions are rolled back instead )
DISADVANTAGE : Starvation possible ( a transaction may keep getting rolled back ), Overhead of maintaining timestamps, More rollbacks compared to locking



Q.) Explain Log-based recovery with example

A.) Log-based recovery is a technique used to restore a database to a consistent state after a system failure ( such as a crash or power failure ). Every operation performed by a transaction is recorded in a log file before it is applied to the database. The log file is stored on stable storage so it survives failures.

Each log record contains :
- Transaction ID
- Data item name
- Old value ( before the operation )
- New value ( after the operation )

Types of Log Records :

1.) < Ti , start > — Transaction Ti has started.
2.) < Ti , X , old_value , new_value > — Transaction Ti has written data item X, changing it from old_value to new_value.
3.) < Ti , commit > — Transaction Ti has committed successfully.
4.) < Ti , abort > — Transaction Ti has been rolled back.

Recovery Operations :

1.) REDO ( Ti ) : If the log has both < Ti , start > and < Ti , commit >, but the changes are not yet in the database → all of Ti's operations are redone ( re-applied ) to ensure the committed data is saved.

2.) UNDO ( Ti ) : If the log has < Ti , start > but no < Ti , commit > → all of Ti's operations are undone ( reversed using old values ) to restore the database to the state before Ti started.

Example :


< T1 , start >
< T1 , A , 500 , 400 >       → T1 wrote A (debited 100)
< T1 , B , 200 , 300 >       → T1 wrote B (credited 100)
< T1 , commit >

< T2 , start >
< T2 , C , 1000 , 800 >      → T2 wrote C (debited 200)
                              → System CRASHES here


After recovery :
- T1 → has commit record → REDO T1 ( re-apply changes to A and B )
- T2 → no commit record → UNDO T2 ( restore C back to 1000 )

ADVANTAGE : Reliable recovery after failures, Supports both undo and redo, Durable commits ( satisfies Durability in ACID )
DISADVANTAGE : Log file can grow very large, Overhead of writing every operation to log, Recovery process can be slow for large logs
